Resumen General de la Versión

Esta actualización integra formalmente la plataforma PediGrido en el módulo de gestión de sucursales, permitiendo el control de apertura y cierre desde el frontend. Además, se implementa una optimización crítica en el dashboard de monitoreo, migrando el cálculo de latencias de un modelo de procesamiento individual a uno basado en datos pre-agregados, lo que mejora significativamente el rendimiento de la aplicación.

Análisis Técnico Detallado

app/src/app/pages/management/branches/branch/branch.component.ts

Análisis General

Se integra internamente la nueva plataforma PediGrido en la configuración de sucursales. Esto permite que el componente reconozca el ID de plataforma 7 y habilite las opciones de gestión para esta nueva integración.

Cambios Detallados

Integración de PediGrido en la Gestión de Sucursales

Explicación: Se expande el arreglo platformsOpenClosed y el objeto platformMap para incluir a PediGrido (ID 7). Este cambio de configuración es necesario para que la interfaz muestre y gestione el estado de conexión de este canal de venta.

Código Antes: 

public platformsOpenClosed = ["Rappi","UberEats","PedidosYa"]
public platformMap = {
  1: 'PedidosYa',
  2: 'Rappi',
  4: 'UberEats',
};
Código Después: 

public platformsOpenClosed = ["Rappi","UberEats","PedidosYa","PediGrido"]
public platformMap = {
  1: 'PedidosYa',
  2: 'Rappi',
  4: 'UberEats',
  7: 'PediGrido' // Nueva plataforma integrada
};

Conclusión del análisis

Expansión Operativa: Habilita el soporte frontend para PediGrido, tratándolo con la misma lógica operativa que a los otros "players" del mercado.

app/src/app/pages/monitoring/latencias/latencias.component.ts

Análisis General

Se refactoriza el cálculo de latencias para utilizar datos ya promediados y agrupados provenientes del backend, reemplazando el procesamiento iterativo de órdenes individuales.

Cambios Detallados

Optimización del Cálculo de Latencias (Datos Pre-Agregados)

Explicación: Se reemplaza el cálculo manual diffMs por una operación basada en los campos latencyTimestamp_avg y totalOrders. Esto reduce la carga computacional en el cliente al utilizar métricas ya procesadas por el servidor.

Cambio en el Eje Temporal de Análisis

Explicación: La fecha de referencia cambia de entryDate (orden individual) a avg_calculatedAt (fecha del cálculo del promedio), asegurando consistencia en la agrupación temporal.

Código Antes: 

const d = this.normalizeDate(orderTime.latencyTimestamp.entryDate);
// ...
b['platformDate-entryDate'] += this.diffMs(
  orderTime.latencyTimestamp.platformDate,
  orderTime.latencyTimestamp.entryDate
);
Código Después: 

const d = this.normalizeDate(orderTime.avg_calculatedAt);
// ...
b['platformDate-entryDate'] += orderTime.latencyTimestamp_avg.platformDate_entryDate * orderTime.totalOrders;
// ... (cálculo optimizado para otras métricas)

Conclusión del análisis

Eficiencia: El frontend evoluciona de procesar miles de registros individuales a sumarizar lotes pre-calculados, mejorando drásticamente el rendimiento.
Escalabilidad: Prepara el dashboard para manejar grandes volúmenes de datos históricos sin latencia en la interfaz.
