Análisis General

Esta actualización, como parte de una **Mejora de Filtros de Búsqueda**, introduce una nueva capacidad de filtrado en la función `search`. El propósito principal de este cambio es permitir a los usuarios segmentar las búsquedas de novedades (órdenes) basándose en si un pedido contiene o no un "canje" .

El impacto funcional es que se añade una nueva dimensión de análisis a la herramienta de búsqueda. Los usuarios ahora pueden responder a preguntas de negocio más específicas, como "mostrar todas las órdenes que no incluyeron un canje" o "mostrar únicamente las órdenes que sí fueron un canje". Esto se logra introduciendo un nuevo parámetro de entrada, `exchangeBoolean`, y construyendo una consulta a MongoDB que filtra los documentos basándose en la existencia y el valor del campo `order.details.canje`.

Cambios Detallados

1. Lectura del Nuevo Parámetro `exchangeBoolean`

Explicación: Se ha modificado la función `search` para que acepte un nuevo parámetro opcional, `exchangeBoolean`, desde el cuerpo de la solicitud. Este parámetro actuará como un interruptor de tres estados para el nuevo filtro de canje.

Código Anterior: 

// ANTES: La función no leía ni procesaba un parámetro para filtrar por canje.
const search = async (req, res) => {
  // ...
  const searchSKU = req.body.searchSKU ? req.body.searchSKU : '';
  const turboBoolean = req.body.turboBoolean ? req.body.turboBoolean : '';
  let cloudBoolean= undefined;
  // ...
};
Código Nuevo: 

// DESPUÉS: Se lee y se inicializa el nuevo parámetro 'exchangeBoolean'.
const search = async (req, res) => {
  // ...
  const searchSKU = req.body.searchSKU ? req.body.searchSKU : '';
  const turboBoolean = req.body.turboBoolean ? req.body.turboBoolean : '';
  // Se lee el nuevo parámetro 'exchangeBoolean' del cuerpo de la solicitud.
  const exchangeBoolean = req.body.exchangeBoolean ? req.body.exchangeBoolean : '';
  let cloudBoolean= undefined;
  // ...
};
2. Lógica de Consulta de MongoDB

Explicación: Se ha añadido una nueva condición ternaria al objeto de consulta de MongoDB. Esta lógica se activa si `exchangeBoolean` tiene un valor diferente de "all" o vacío.

  • Si `exchangeBoolean` es `"noExchange"`, construye una consulta `$or` que busca documentos donde el campo `order.details.canje` no existe, es `null` o es `0`.
  • Si `exchangeBoolean` tiene cualquier otro valor (implícitamente, `"exchange"`), construye una consulta que busca documentos donde `order.details.canje` es `1`.
Código Anterior: 

// ANTES: El objeto de consulta 'data' no incluía un filtro por 'canje'.
// ...
  const data = {
    // ...
    ...(searchSKU != '' ? { 'order.details.sku': { $in: [searchSKU, parseInt(searchSKU)]}  } : {}),
    ...((turboBoolean != 'all' && turboBoolean != '') ? { 'order.delivery_operation_type': turboBoolean } : {})
  };
// ...
Código Nuevo: 

// DESPUÉS: Se añade la lógica condicional para filtrar por 'canje'.
// ...
  const data = {
    // ...
    ...(searchSKU != '' ? { 'order.details.sku': { $in: [searchSKU, parseInt(searchSKU)]}  } : {}),
    ...((turboBoolean != 'all' && turboBoolean != '') ? { 'order.delivery_operation_type': turboBoolean } : {}),
    // Se añade la nueva lógica de filtrado para 'exchangeBoolean'.
    ...((exchangeBoolean != 'all' && exchangeBoolean != '') ?
      // Si el valor es 'noExchange', se busca por la ausencia de un canje.
      exchangeBoolean === 'noExchange' ? {
        $or: [
          {'order.details.canje' : { $exists: false }}, // El campo no existe.
          {'order.details.canje' : 0 },                 // El campo es 0.
          {'order.details.canje' : null }                // El campo es nulo.
        ]
      // Si el valor es otro (ej. 'exchange'), se busca por la presencia de un canje.
      } : {'order.details.canje' : 1 } : {})
  };
// ...

Conclusión del análisis

Este commit introduce una mejora en la capacidad de filtrado del endpoint de búsqueda de novedades. La modificación permite a los consumidores de la API segmentar los resultados entre pedidos que son parte de un "canje" y los que no lo son.

  • Prepara para el Frontend: Este cambio enriquece la API, dándole más flexibilidad al frotEnd.
