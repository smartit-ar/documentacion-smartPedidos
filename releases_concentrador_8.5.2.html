Resumen General de la Versión

Esta actualización representa una mejora técnica enfocada en la calidad del dato y la robustez del sistema de monitoreo. Se refina la lógica de consulta para asegurar que solo se procesen registros con tipos de datos válidos (evitando inconsistencias por valores nulos o vacíos) tanto en el endpoint de monitoreo como en el proceso CRON diario. Además, se añade trazabilidad detallada en el sistema de notificaciones por correo.

Análisis Técnico Detallado

api/src/controllers/monitoring.js

Análisis General

Se refina la lógica de filtrado para los tiempos de pedido. La funcionalidad ahora es más estricta, garantizando que los datos devueltos para el análisis de performance sean válidos y consistentes.

Cambios Detallados

Refinamiento en el Filtrado de Latencias ($type vs $exists)

Explicación: Se ha reemplazado el operador $exists: true por $type: ["date", "string"]. Esto asegura que el sistema ignore registros que, aunque existan, contengan valores inválidos (null, false) que podrían romper los cálculos matemáticos de latencia.

Código Antes: 

filters['latencyTimestamp.platformDate'] = { $exists: true };
    filters['latencyTimestamp.responseDate'] = { $exists: true };
    filters['latencyTimestamp.sendSqsDate'] = { $exists: true };
    filters['latencyTimestamp.receiveSqsDate'] = { $exists: true };
    filters['latencyTimestamp.finalResponseDate'] = { $exists: true };
Código Después: 

filters['latencyTimestamp.platformDate'] =  { $type: ["date", "string"] }; 
    filters['latencyTimestamp.responseDate'] =  { $type: ["date", "string"] }; 
    filters['latencyTimestamp.sendSqsDate'] =  { $type: ["date", "string"] }; 
    filters['latencyTimestamp.receiveSqsDate'] = { $type: ["date", "string"] }; 
    filters['latencyTimestamp.finalResponseDate'] = { $type: ["date", "string"] };

Conclusión del análisis

Robustez: El endpoint de monitoreo es ahora resiliente frente a inconsistencias de tipos de datos en la base de datos.
