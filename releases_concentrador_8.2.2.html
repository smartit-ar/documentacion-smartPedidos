Resumen General de la Versión

Este release contiene una corrección fundamental en el controlador de sucursales (`branch.js`). Se refactoriza la lógica utilizada para identificar al "tenant" correcto asociado a un pedido de MercadoPago. Esto se realiza implementando un flujo de búsqueda robusto que utiliza el `user_id` de MercadoPago como clave, eliminando la dependencia en el nombre de la sucursal, lo que asegura una identificación precisa y sin ambigüedades.

api/src/controllers/branch.js Commit ead77710bffa42b763dc1aad38d3ac7a90f6fc92

Análisis General

Esta actualización, como parte de una **Mejora en la Identificación de Tenant para MercadoPago**, consiste en una refactorización crucial dentro de la función `sendReject`, específicamente en el bloque de código que maneja las confirmaciones para MercadoPago. El propósito principal de este cambio es corregir y robustecer la lógica utilizada para identificar al "tenant" correcto asociado a un pedido.

Anteriormente, la lógica intentaba encontrar al tenant basándose en el nombre de la sucursal, un método que resultaba impreciso y propenso a errores. La nueva implementación adopta un enfoque mucho más fiable: utiliza el `user_id` de MercadoPago (asociado al pedido) para buscar la sucursal en la base de datos, luego obtiene la "Cadena" (Chain) a la que pertenece esa sucursal y, finalmente, utiliza el nombre de la cadena para identificar al tenant. Este cambio asegura una correspondencia unívoca y correcta entre el pedido de MercadoPago y la configuración del tenant correspondiente.

Cambios Detallados

Explicación: Se ha reescrito por completo la forma en que se determina el `tenant` para un pedido de MercadoPago. El flujo ahora sigue una secuencia lógica más robusta: **user_id del pedido -> Sucursal -> Cadena -> Tenant**. Esto desacopla la identificación del tenant del nombre de la sucursal, que puede no ser un identificador único o fiable para este propósito.

    // 1. Se obtiene el 'user_id' de MercadoPago desde el principio, ya que es el identificador clave.

Conclusión del análisis

La refactorización en `api/src/controllers/branch.js` es una corrección crítica que soluciona un fallo fundamental en la lógica de negocio para MercadoPago.

  • **Precisión en la Identificación:** El sistema ahora utiliza un flujo de datos fiable (`user_id` -> Sucursal -> Cadena -> Tenant) para identificar al tenant, en lugar de depender de un campo propenso a errores como el nombre de la sucursal.
  • **Robustez:** Se han añadido múltiples puntos de validación y logging a lo largo del nuevo flujo (verificación de `user_id`, `currentBranch`, `chainObj`), lo que facilita enormemente la depuración de futuros problemas.
  • **Correctitud Funcional:** Este cambio asegura que, para cada pedido de MercadoPago, se utilicen las credenciales y configuraciones del tenant correcto, lo cual es indispensable para el funcionamiento de la integración.

Análisis General

Esta actualización, como parte de la iniciativa **TENANT**, se centra en eliminar un valor "hardcodeado" (fijo en el código) crítico en la integración con MercadoPago y reemplazarlo por una lógica dinámica. El objetivo principal es asegurar que cada solicitud enviada a la API de "Cloud" para obtener el token de MercadoPago contenga el **TenantId** correcto, basado en la cadena a la que pertenece la sucursal.

Para lograr esto, se ha refactorizado la lógica de identificación del tenant dentro de las funciones `sendReject` y `rejectOrdersPeya`. Se ha introducido un flujo de búsqueda más robusto que utiliza el **user_id** del pedido de MercadoPago para identificar la sucursal y, a partir de ahí, su cadena y tenant correspondientes. Este cambio es fundamental para el correcto funcionamiento de la integración en un entorno multi-tenant, donde cada cadena (tenant) tiene su propio identificador.

1. Eliminación de TenantId Hardcodeado y Uso Dinámico

api/src/models/branch.js Commit 4726dfc1c050dde8208efd94fbfb8c75fb2b3449

Análisis General

Esta actualización, como parte de la iniciativa **TENANT** y la integración con MercadoPago, modifica el esquema de la base de datos para las sucursales (branchSchema). El cambio consiste en añadir un nuevo campo, **mercadoPagoUserId**, a la configuración de cada plataforma asociada a una sucursal.

Este ajuste en el modelo de datos es un **prerrequisito técnico indispensable** para la refactorización realizada en el controlador `branch.js`. Al proporcionar un campo dedicado para almacenar el ID de usuario de MercadoPago a nivel de sucursal, se habilita la nueva lógica de búsqueda que identifica de manera unívoca una sucursal a partir de un `user_id` proveniente de un pedido. Esto robustece significativamente el proceso de identificación del tenant correcto para cada transacción de MercadoPago.

Cambios Detallados

Explicación: Se ha añadido un nuevo campo llamado `mercadoPagoUserId` de tipo `Number` al sub-documento dentro del array `platforms`. Este campo está diseñado para almacenar el identificador de usuario específico de MercadoPago que está asociado a esa sucursal para esa integración en particular.

Código Anterior:

      }

Código Nuevo:

Conclusión del análisis

La modificación en el esquema `branchSchema` es un cambio estructural que habilita la nueva lógica de negocio para MercadoPago.

  • **Identificación Unívoca:** Proporciona el soporte en la base de datos necesario para que el controlador pueda realizar una búsqueda fiable de la sucursal a partir del `user_id` de un pedido, como se implementó en la función `sendReject`.
  • **Desacoplamiento:** Ayuda a desacoplar la lógica de negocio de identificadores menos fiables (como el nombre de la sucursal), basando la identificación en un dato concreto y único proporcionado por la plataforma externa.
  • **Habilitador de Funcionalidad:** Este cambio es el habilitador directo que permite que la refactorización de la búsqueda de tenant en el controlador funcione correctamente.
