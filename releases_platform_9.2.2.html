Resumen General de la Versión

Este release se enfoca en la estandarización de tipos de datos a nivel de las interfaces (parsers) de las plataformas. Se asegura que el identificador original de cada pedido (`originalId`) sea siempre almacenado como una cadena de texto (`String`), lo cual previene inconsistencias en la base de datos y errores sutiles en las operaciones de búsqueda y comparación en capas posteriores del sistema.

Análisis Técnico Detallado

api/src/platforms/interfaces/mercadoPago.js, rappi.js, thirdParty.js, uberEats.js

Análisis General

Este commit aplica una **corrección de estandarización de tipo de dato** de manera consistente a través de todos los parsers (interfaces) de las plataformas. El cambio es puntual pero fundamental para garantizar la integridad y consistencia de los datos en el sistema.

El objetivo es asegurar que el campo **`originalId`**, que almacena el identificador del pedido de la plataforma externa, sea **siempre de tipo `String`**, independientemente del tipo de dato (número o texto) que envíe la plataforma de origen.

Cambios Detallados

1. Unificación del Tipo de Dato para order.originalId

Explicación: En los cuatro archivos modificados, se aplica la misma corrección dentro de la función `orderMapper`. Se fuerza la conversión del ID del pedido a `String` usando el método `.toString()` antes de asignarlo a la propiedad `order.originalId`.

Código Anterior (Patrón Genérico): 

// ANTES: El código asignaba el ID del pedido directamente. Si la plataforma enviaba un ID numérico (ej: 98765),
// el campo 'originalId' se guardaba como un número, creando inconsistencias.
// ...
              let order = {};
              order.id = data.id; // o data.posId
              order.originalId = data.id; // o data.originalId
              order.displayId = data.id; // o data.displayId
// ...
Código Nuevo (Patrón Genérico): 

// DESPUÉS: El código ahora fuerza la conversión a string en todos los casos.
// ...
              let order = {};
              order.id = data.id; // o data.posId
              order.originalId = data.id.toString(); // o data.originalId.toString()
              order.displayId = data.id; // o data.displayId
// ...
Ejemplos de Aplicación por Archivo

// api/src/platforms/interfaces/mercadoPago.js
order.originalId = data.id.toString();

// api/src/platforms/interfaces/rappi.js
order.originalId = data.originalId.toString();

// api/src/platforms/interfaces/thirdParty.js
order.originalId = data.originalId.toString();

// api/src/platforms/interfaces/uberEats.js
order.originalId = data.id.toString();

Conclusión del análisis

Este commit es una mejora de robustez y una buena práctica de programación defensiva.

  • **Consistencia de Datos:** Al forzar la conversión a `String` en el punto de entrada de los datos (los parsers de cada plataforma), se neutraliza el riesgo de inconsistencias de tipo entre los identificadores de pedidos.
  • **Prevención de Errores:** Previene errores sutiles en capas más profundas del sistema, especialmente en operaciones de búsqueda o comparación que dependen de tipos consistentes.
  • **Integridad y Previsibilidad:** Asegura que los datos se almacenen de manera uniforme en la base de datos y hace que el comportamiento del sistema sea más predecible.
