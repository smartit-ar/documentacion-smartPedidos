Documentación: Reglas de Negocio por Cadena y Plataforma

1. Propósito y Objetivo

Esta funcionalidad introduce un sistema para definir reglas de negocio específicas que se aplican a la combinación de una Cadena (chain) y una Plataforma (platform). El objetivo es permitir una configuración granular y flexible (ej: tiempos de preparación, auto-aceptación de pedidos) sin modificar el código principal, desacoplando la lógica de negocio de la implementación de las integraciones.

Esto permite que, por ejemplo, la cadena "Grido" tenga un tiempo de preparación de 15 minutos para "PedidosYa", pero de 20 minutos para "Rappi".

2. Arquitectura de Datos: Nueva Colección chainBusinessRules

Para almacenar estas reglas, se ha creado una nueva colección en MongoDB llamada chainbusinessrules. Cada documento en esta colección representa un único "contrato" de reglas entre una cadena y una plataforma.

2.1. Modelo Mongoose (api/src/models/chainBusinessRules.js)

El esquema de Mongoose define la estructura, validaciones y relaciones de los documentos.


import mongoose, { Schema } from 'mongoose';
import uniqueValidator from 'mongoose-unique-validator';

const chainBusinessRulesSchema = new Schema({
    // Referencia a la colección 'chains'
    chainId: {
        type: Schema.Types.ObjectId,
        ref: 'chain',
        required: true
    },
    // Referencia a la colección 'platforms'
    platformId: {
        type: Schema.Types.ObjectId,
        ref: 'platform',
        required: true
    },
    // Objeto anidado que contiene las reglas
    rules: {
        autoAccept: {
            type: Boolean,
            default: false,
            required: true
        },
        preparationTime: {
            type: Number,
            enum: [5, 10, 15, 20, 25, 30],
            required: true
        }
    }
}, {
    timestamps: true, // Gestiona createdAt y updatedAt
    strict: false
});

// Índice único compuesto para prevenir duplicados
chainBusinessRulesSchema.index({ platformId: 1, chainId: 1 }, { unique: true });

// Plugins para validación y formato JSON
chainBusinessRulesSchema.plugin(uniqueValidator);
chainBusinessRulesSchema.plugin(require('@meanie/mongoose-to-json'));

module.exports = mongoose.model('chainBusinessRules', chainBusinessRulesSchema);

3. API para la Gestión de Reglas (CRUD)

Se han expuesto nuevos endpoints para permitir la gestión completa (Crear, Leer, Actualizar, Eliminar) de estas reglas desde el Backoffice.

3.1. Controlador (api/src/controllers/chainBusinessRules.js)

Este controlador orquesta las operaciones CRUD. Reutiliza los helpers genéricos para mantener la simplicidad y consistencia.


import model from '../models/chainBusinessRules';
import _helpers from './_helpers';

// Encuentra todas las reglas (con filtros opcionales)
const findAll = (req, res) => _helpers.find(model, req, res);

// Encuentra una regla por su ID
const findById = (req, res) => _helpers.findById(model, req, res);

// Crea una nueva regla
const saveOne = (req, res) => _helpers.save(model, req, res);

// Actualiza una regla existente
const updateOne = (req, res) => _helpers.findByIdAndUpdate(model, req, res);

// Elimina una regla existente (eliminación física)
const deleteOne = (req, res) => _helpers.destroyModel(model, req, res);

module.exports = { findAll, findById, saveOne, updateOne, deleteOne };

3.2. Rutas (api/src/routes/chainBusinessRules.js)

Este archivo define los endpoints y los asocia a las funciones del controlador.


import express from 'express';
const router = express.Router();
import controller from '../controllers/chainBusinessRules';

// Rutas para listar todas las reglas y crear una nueva
router.route('/')
  .get(controller.findAll)
  .post(controller.saveOne);

// Rutas para obtener, actualizar o eliminar una regla específica por su ID
router.route('/:id')
  .get(controller.findById)
  .put(controller.updateOne)
  .delete(controller.deleteOne);

module.exports = router;

A continuación se muestran ejemplos de cómo interactuar con los nuevos endpoints.

4.1. Crear una Nueva Regla

POST /api/chainBusinessRules

Propósito: Crear la regla para "Grido" y "PedidosYa".

Request Body (JSON):


{

  "platformId": "a24d12527267ecfd0b946390",
  "rules": {
    "autoAccept": true,
    "preparationTime": 15
  }
}

{

4.3. Actualizar una Regla Existente

PUT /api/chainBusinessRules/6887bbc2876ae6b43efc4fa4

Propósito: Cambiar el preparationTime de 15 a 20 para la regla Grido-PedidosYa.

4.4. Eliminar una Regla

DELETE /api/chainBusinessRules/6887bbc2876ae6b43efc4fa4

Propósito: Eliminar la regla de negocio entre Grido y PedidosYa. (Respuesta 204 No Content)

Impacto a Nivel de Sucursal (branch)

Un cambio fundamental es que las reglas de negocio, una vez definidas, se incrustan directamente en el documento de la sucursal (branch) correspondiente. Esto optimiza las consultas cada vez que se necesita acceder a las reglas para una plataforma específica de una sucursal.

El documento de la sucursal ahora contiene un objeto rules dentro de cada elemento del array platforms.
