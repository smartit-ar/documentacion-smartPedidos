Resumen General de la Versión

Esta actualización implementa un sistema robusto de métricas de latencia para el ciclo de vida de las órdenes ("Order Lifecycle Tracing"). Se reestructuró el modelo de datos y se instrumentaron controladores, estrategias de cola y gestores de plataforma para capturar hitos temporales críticos: recepción (entry), encolado (sendSqs), consumo (receiveSqs) y confirmación final (finalResponse). Esto permite medir y diagnosticar cuellos de botella en cada etapa del procesamiento.

Análisis Técnico Detallado

api/src/models/orderTime.js

Análisis General

Se redefine el esquema de datos para agrupar las métricas temporales bajo un subdocumento latencyTimestamp. Esto organiza mejor la información y separa los metadatos de la orden de los hechos temporales.

Cambios Detallados

Estructura Jerárquica y Timestamps

Explicación: Se introduce el campo latencyTimestamp de tipo Object. Se activan los timestamps: true de Mongoose para tener auditoría de creación del registro.

Código Nuevo: 

// DESPUÉS: Esquema actualizado
    latencyTimestamp: {
       type: Object
    }
}, {
    strict: false,
    timestamps: true
});

api/src/controllers/ (mercadoPago.js, peya.js, etc.)

Análisis General

Se realiza una instrumentación completa de los controladores de entrada para registrar los tiempos de latencia inicial ("Controller Latency").

Cambios Detallados

Registro de Tiempos de Latencia

Explicación: Al finalizar el procesamiento exitoso, se crea un registro en la colección orderTime capturando entryDate y responseDate. En integraciones asíncronas, también se persiste el sendSqsDate.

Código Nuevo (Ejemplo MercadoPago): 

// DESPUÉS: Registro de hitos de entrada y salida del controlador
    await orderTimeModel.create({
      entryDate: entryDate,
      responseDate: new Date(),
      branch: currentBranch.branchId,
      order: orderId,
      platform: "MercadoPago"
    })

api/src/platforms/management/platform.js

Análisis General

Se instrumenta el punto crítico de comunicación asíncrona: el envío a la cola SQS. Esto permite medir la latencia interna de procesamiento antes de delegar la orden al sistema POS.

Cambios Detallados

Captura de Hito de Encolado (sendSqsDate)

Explicación: Se añade await a la llamada de encolado para garantizar la entrega a AWS y se registra inmediatamente el timestamp.

Código Nuevo: 

// DESPUÉS: Espera explícita y marca de tiempo
            await this.aws.pushAutoReplyToQueue(...);
            orderProccessed.sendSqsDate = new Date();

api/src/platforms/management/strategies/orderStrategy/receiveStrategy.js

Análisis General

Se modifica la estrategia que orquesta la recepción de órdenes desde la cola para capturar el inicio del consumo del mensaje.

Cambios Detallados

Inyección de dateSQS

Explicación: Se captura un timestamp justo antes de invocar a la plataforma, representando el receiveSqsDate. Este dato se pasa a la implementación específica para su persistencia posterior.

Código Nuevo: 

// DESPUÉS: Captura del inicio de consumo de la orden
        let dateSQS = new Date();
        // ...
        const platformResult = await this.platform.receiveOrder(
          this.savedNew.order,
          dateSQS // Se pasa el timestamp a la implementación específica
        );

api/src/platforms/management/platform/*.js (Implementaciones)

Análisis General

Se actualizan los métodos receiveOrder de las distintas plataformas para cerrar el ciclo de trazabilidad, registrando la confirmación final de negocio.

Cambios Detallados

Actualización de finalResponseDate

Explicación: Se utiliza orderTimeModel.updateOne para inyectar la fecha de confirmación final en el documento existente, indicando el fin del procesamiento lógico.

Código Nuevo: 

// DESPUÉS: Registro del fin del procesamiento de negocio
 await orderTimeModel.updateOne(
   { order: orderId },
   { $set: { 'latencyTimestamp.finalResponseDate': new Date() } }
 );

Conclusión del análisis

Network Latency: Calculable entre entryDate y responseDate.
Internal Processing: Calculable entre sendSqsDate y receiveSqsDate.
External Integration: Calculable entre receiveSqsDate y finalResponseDate.
